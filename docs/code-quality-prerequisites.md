# 代码质量前提（落码前必读）

## 定位

本文件是实现类工作的**硬前提**。  
用户要的是可长期演进的高质量代码，不是门禁报表、任务状态机仪式，或“Blocked / DevTools 不可用”类过程说明。

除非用户**明确要求**交付编排、任务状态推进、平台验收或流程门禁，否则默认：

- **不**主动管理任务 front matter / issue 状态机 / 门禁通过与否
- **不**把时间花在解释为什么无法做微信 DevTools、真机、浏览器人工验收
- **只**围绕需求写出高质量、可维护、可扩展、健壮的代码，并用可运行的本地验证证明行为正确

## 落码前检查清单（必须完成，再写代码）

按顺序自检；任一项说不清时，先读代码或补清边界，再动手。

### 1. 意图与验收

- [ ] 用户本轮明确要什么（行为结果，不是文件列表）
- [ ] 可观察验收点 ≤ 3 个主目标；邻近优化记 follow-up，不塞进本轮
- [ ] 与 `docs/requirements.md`、相关 task/OpenAPI/现有行为冲突时，以**用户最新意图 + 既有契约**对齐，避免 silent break

### 2. 现状基线（短基线，禁止全仓漫游）

- [ ] 已读将改模块及其直接依赖的现有模式（命名、分层、错误处理、测试风格）
- [ ] 已确认单一事实来源：API 契约、域规则、客户端状态归属（避免第二套真相）
- [ ] 已标出不应触碰的范围（无关页面、无关域、用户未提及的重构）

### 3. 设计边界（质量 / 可维护 / 可扩展）

- [ ] **职责清晰**：展示 / 编排 / 域规则 / 数据访问 / 平台适配分层清楚；不把业务规则塞进 UI 或 HTTP 胶水
- [ ] **复用优先**：沿用现有 API client、域模块、纯函数工具、状态封装；没有第二消费者不抽共享层
- [ ] **扩展点克制**：为真实变化预留边界（接口、谓词、状态机、DTO 映射），不做空想配置化/过度抽象
- [ ] **契约稳定**：对外 HTTP/DTO/错误码变更有意识；破坏性变更需兼容策略或显式迁移
- [ ] **依赖方向**：内层不依赖外层 UI/平台 API；全局状态有明确 owner

### 4. 健壮性

- [ ] 非法输入、权限、空数据、并发/重复提交、幂等、过期会话、网络失败均有明确行为
- [ ] 错误不吞：有用户可读信息或可观测日志；不把失败伪装成成功
- [ ] 边界条件有测试或等价本地验证（至少覆盖主路径 + 1 条关键失败路径）
- [ ] 安全/隐私：不泄露内部 ID、密钥、精确隐私字段；鉴权在服务端生效

### 5. 实现纪律

- [ ] 最小 diff：只改与目标相关的代码
- [ ] 命名、结构、注释风格与仓库一致；不为“显得工程化”加样板
- [ ] 改完跑最近模块的测试/语法检查；失败先修根因，不降断言

## 质量标准（写代码时持续对照）

| 维度 | 要求 |
| --- | --- |
| **正确性** | 行为匹配意图与契约；测试或本地命令可复现 |
| **可维护** | 读者 5 分钟内能找到状态 owner、数据流和失败出口；无神秘副作用 |
| **可扩展** | 新字段/新状态/新入口能局部扩展，不需重写整页或整域 |
| **健壮** | 异常路径一等公民；幂等、权限、空态、重试可预期 |
| **简洁** | 能用一个清晰函数解决的不拆三层；删除重复，不堆 indirection |
| **一致** | 同问题同模式（错误码、request-state、域谓词、DTO 映射） |

## 落码后（轻量，默认不做门禁运营）

1. 用最近的模块命令验证（如 `npm test`、`node --test`、相关 syntax check）。
2. 自查 diff 是否越界。
3. 向用户说明：**改了什么行为、为何这样设计、如何验证**。
4. **不要**默认更新 task 状态、Blocked 原因、门禁清单，除非用户要求交付/任务推进。

## 与交付工作流的关系

- `docs/delivery-workflow.md` 仅在用户明确使用 `交付` / 任务状态 / 多角色编排时适用。
- 即使用户说“交付”，实现核心仍是本文件的质量前提；流程是辅助，不是目标。
- 平台人工验收（DevTools/真机/浏览器）由用户自行决定是否做；代理默认不阻塞、不反复汇报这类门禁。
