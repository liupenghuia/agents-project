# Iterative Implementation Guidelines

## Purpose

在重复修改、局部优化和跨角色接力中，优先完成用户明确要求，并尽量保留当前已经可用的行为。视觉、结构或交互上的“不理想”如果没有明确验收依据，应记录为后续优化，不应在本轮擅自扩大范围。

落码前必须先满足 `docs/code-quality-prerequisites.md`：默认不管流程门禁，只保证代码质量、可维护性、可扩展性与健壮性。

## Context Compression Protocol

每次开始工作时只建立一份短基线，包含：

- 当前任务、验收标准、状态和阻塞条件。
- 用户本轮新增或改变的意图。
- 已确认可用的行为、接口和页面。
- 本轮允许修改的文件和不应触碰的范围。
- 最近一次验证命令、结果和未解决风险。

基线完成后，后续搜索优先围绕目标符号、路由、测试和相关历史记录进行，不重复加载无关模块或整份文档。

## Change Discipline

1. 先判断请求属于修复、需求实现还是后续优化。没有明确授权时，不把优化改写成重构。
2. 修改前列出最多三个可观察结果；每个结果必须能通过测试、页面状态或接口响应验证。
3. 先使用现有模式、组件、API client、状态管理和样式；只有现有模式无法满足验收时才增加抽象。
4. 单轮只处理一个主要行为主题。发现邻近问题时记录路径和证据，不顺手扩大修改。
5. 保留用户已经做出的修改；无法确定归属时以最小差异方式合并，不覆盖或重写整个文件。
6. 对未确认的视觉偏好使用保守改动，并在交接记录中标注“待优化”，而不是把猜测写成完成标准。

## Fast Verification Loop

每轮修改按以下顺序闭环：

1. 读取目标文件和直接依赖，确认现状。
2. 做最小实现，避免顺手整理无关代码。
3. 运行文件级语法/单元测试。
4. 运行任务级 runner 或最接近的验证命令。
5. 检查 diff，确认没有越界改动。
6. 用户明确要求交付/任务推进时，才更新任务状态与 handoff；默认只回报工程验证结果。

测试失败时只修复当前失败的根因。缺失环境、凭据或平台工具时，不假装通过，也不把本轮变成门禁讨论；继续保证仓库内可验证部分的代码质量。

## Handoff Format

每次交接只保留以下信息：

- 已完成的可观察行为。
- 精确变更文件。
- 精确命令和结果。
- 尚未确认的内容及原因。
- 下一轮唯一最高优先级动作。

“看起来不对”必须转换为具体描述，例如“返回页面后筛选条件丢失”“按钮在 320px 宽度下换行”“取消拉黑后列表未恢复”。无法具体描述的问题暂不扩大实现范围。

## Completion Boundary

满足当前验收标准即可完成当前编码轮，不要求一次性达到最终视觉或架构理想状态。后续优化应创建独立任务或 issue，包含目标、当前行为、期望行为和复现证据；这样下一轮只需读取该记录，不必重新推断整个上下文。
