# Product Request Template

这是非技术产品负责人提交需求的默认入口。产品负责人只需要描述目标和用户结果，不需要指定代码方案。

## 默认输入

```text
请帮我完成这个产品需求：

[用自己的话描述需求]

我的目标是：
[我希望用户最终能做到什么]
```

## 顺序完成指令

当产品负责人发送“顺序完成”时，按以下顺序执行：

1. Product Agent 整理需求、范围、用户流程和验收标准。
2. 发现重要产品取舍时，用非技术语言说明，并只询问必要决策。
3. Architect Agent 设计前后端边界、数据模型、API、权限、隐私和回滚方案。
4. 根据依赖关系依次实现后端和前端；可并行的范围只有在合同稳定后才并行。
5. 补充正常流程、加载、空数据、错误、重试、权限、登录失效和隐私处理。
6. 运行适用的单元、集成、客户端和交付验证。
7. Test Agent 独立对照验收标准检查，不替实现 Agent 隐藏失败。
8. 对失败项创建或更新 issue，由所属 Agent 修复，再由 Test Agent 独立复测。
9. 未验证的内容必须标记为待验证、Blocked 或未完成，不得报告为完成。
10. 最终用简单语言汇报：已完成什么、尚未完成什么、是否可以体验、产品负责人下一步只需做什么。

## 固定约束

- 不擅自扩大需求或改变用户目标。
- 不重写无关模块，不破坏现有正常功能。
- 不确定的产品行为先记录为假设或未知，不把技术猜测当成产品决定。
- 需要产品取舍时只提出最小必要问题，并说明各选项对用户和进度的影响。
- 生产发布、真实用户数据、密钥、付费外部服务、不可逆操作和敏感权限必须先获得产品负责人的明确确认。
- 发现后续优化时建立独立记录，不在当前任务中顺手扩大范围。

## 产品负责人关注点

产品负责人只需要确认：

- 目标用户是否正确。
- 用户最终是否能完成目标。
- 哪些功能必须现在完成。
- 哪些功能可以后续优化。
- 验收结果是否符合预期。

产品负责人不需要决定：

- 使用什么框架或数据库。
- API 如何实现。
- 文件如何拆分。
- 测试如何编写。

这些内容由 Architect、Backend、Frontend 和 Test Agent 在既定边界内负责。
