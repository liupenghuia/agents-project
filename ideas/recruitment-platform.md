# 产品简报：招聘与应聘双向平台

## 原始想法

做一个招聘和应聘的双向平台，支持小程序和 App。新用户进入应用后选择“招人”或“应聘”，进入对应端开始注册；注册完成后角色确定。当前第一阶段先做角色选择和注册模块。

## 一句话定位

面向需要发布招聘需求的招人方和寻找工作机会的应聘方，提供一个从首次进入就明确角色、分别完成注册并进入对应使用端的双向招聘平台。

## 问题定义

- 目标用户：招人方、应聘方，以及需要在两种角色中选择一类开始使用的新用户。
- 使用场景/触发时机：用户首次打开小程序或 App，准备发布招聘需求或寻找工作。
- 用户遇到的问题或未满足需求：当前尚未明确，需要验证用户是否需要一个同时服务双方但入口清晰的平台。
- 当前解决方式：未知，可能是招聘网站、社交平台、线下中介或熟人介绍；不能在没有证据时当作事实。
- 当前问题造成的成本：待验证，包括找人效率、求职效率、信息分散和角色理解成本。
- 为什么现在解决：用户提出先建设双向招聘平台，具体市场时机待补充。

## 用户任务与结果

- 当用户首次打开平台时，用户想选择自己的使用目的，从而进入不会混淆的招人端或应聘端。
- 当用户选定角色后，用户想用最少必要信息完成注册，从而进入对应的后续功能。
- 期望的用户结果：用户知道自己当前是什么角色，注册资料提交后能看到审核状态，并能在下次打开时进入已审核的身份。
- 期望的业务结果：提高新用户从首次打开到完成注册的转化率，并减少角色混淆造成的流失。

## 价值主张

- 核心价值：用明确的角色入口降低招聘双方的使用复杂度。
- 差异化：同一平台承载双向需求，但从第一次进入就按角色提供独立体验。
- 用户为什么相信我们能做到：第一阶段只承诺角色选择、注册和身份承接，后续招聘撮合能力不提前承诺。

## 核心用户流程

### 招人方身份

1. 新用户打开微信小程序。
2. 看到两个明确入口：“招人”和“应聘”。
3. 点击“招人”。
4. 进入招人方注册流程。
5. 填写招人方最小资料并提交。
6. 创建“招人方”身份，状态为“待人工审核”。
7. 进入审核等待页，展示当前身份和审核状态。

### 应聘方身份

1. 新用户打开微信小程序。
2. 看到两个明确入口：“招人”和“应聘”。
3. 点击“应聘”。
4. 进入应聘方注册流程。
5. 填写应聘方最小资料并提交。
6. 创建“应聘方”身份，状态为“待人工审核”。
7. 进入审核等待页，展示当前身份和审核状态。

### 双身份流程

1. 同一微信用户已有一种身份。
2. 用户从身份入口选择创建另一种身份，而不是修改原身份。
3. 用户填写另一种身份的注册资料并提交人工审核。
4. 两种身份分别保留，进入某一种身份时不能把它转换成另一种身份。

### 必须覆盖的异常流程

- 用户未选择角色就尝试继续。
- 用户选择角色后返回入口页。
- 注册信息为空、格式错误或不完整。
- 微信授权或手机号授权失败、取消或过期。
- 同一身份重复提交注册。
- 网络超时或服务不可用。
- 注册提交后再次打开应用，恢复身份和审核状态。
- 用户已有一个身份后创建另一个身份。
- 审核通过、审核拒绝和被要求修改资料。

## MVP 第一阶段

### 必须包含

- 微信小程序新用户入口。
- 清晰展示“招人”和“应聘”两个角色选择。
- 招人方最小注册流程。
- 应聘方最小注册流程。
- 表单校验、重复提交、失败和成功状态。
- 同一微信用户可以分别创建两种身份。
- 每个身份独立保存，创建后不可直接转换身份类型。
- 注册资料提交后进入人工审核，支持待审核、通过、拒绝/需修改状态。
- 再次进入时识别已有身份和审核状态。
- 微信用户授权、隐私政策和用户协议入口，符合微信小程序规范。
- 可验证的埋点或日志：入口展示、角色选择、注册开始、注册成功、注册失败。

### 明确不做

- 招聘职位发布。
- 简历创建或投递。
- 招聘搜索、匹配和推荐。
- 在线沟通、面试和录用。
- 薪资、支付和会员。
- 招人方企业认证、应聘方实名认证和身份证件采集。
- 注册完成后的完整业务首页，第一阶段只需要明确承接位置或占位页。

### 后续机会

- 招人方发布职位和管理候选人。
- 应聘方完善简历、搜索职位和投递。
- 双向匹配、沟通、面试和录用流程。
- 企业认证、实名认证和信用机制。
- 求职端个人信息填写与求职信息提交。
- 招聘者信息填写和招聘信息发布。
- 双向招聘/求职信息市场、详情、联系方式和收藏。

## 交付假设

- 必需客户端：微信小程序。
- 后端/API 影响：需要微信登录、用户账户、多个身份档案、注册资料、审核状态和登录态恢复。
- 数据/存储影响：同一用户与多个角色档案分离保存；具体模型由 Architect Agent 设计。
- 外部服务：微信登录、用户信息/手机号授权能力，具体使用方式由 Architect 评估。
- 运维或客服影响：需要人工审核、处理拒绝原因、资料修改、重复提交和账号恢复问题。

## 风险与约束

- 产品/用户采用风险：如果用户不理解“招人”和“应聘”的区别，首次选择会产生误导和流失。
- 隐私/安全/合规风险：注册资料、手机号、身份信息和用户协议处理方式尚未确定。
- 技术可行性风险：微信授权、隐私弹窗、手机号能力和审核后台需要满足小程序规范。
- 成本/时间/团队约束：第一阶段只做小程序、角色选择、注册、审核状态和身份承接，不提前实现招聘业务。
- 无障碍/本地化约束：角色入口和注册表单需要支持中文、清晰标签和错误提示。
